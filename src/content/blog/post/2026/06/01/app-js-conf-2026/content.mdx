*My recap of App.js Conf 2026 in Krakow: talks, libraries, the 'elephant in the room' called AI*

---

After [React Universe 2024](/blog/post/2024/10/21/react-universe-2024) in Wrocław, I had the chance to go back to Poland
for another React Native conference. This time the destination was Krakow, and the conference was
[App.js Conf 2026](https://appjs.co), organized by [Software Mansion](https://swmansion.com) together with
[Expo](https://expo.dev). Two days, one track, a packed venue, and a lineup that made it very clear where the React
Native ecosystem is heading.

In this post I want to share the talks that stood out to me the most, the libraries and tools that were announced or
demoed on stage, and a few personal thoughts about the direction of the ecosystem and the cofnerence in general. 

## Some numbers

Before getting into the talks, a bit of context about the event:

**Location:** 12 Świętego Wawrzyńca, 31-060 Kraków
**Dates:** 27-28 May 2026  
**Format:** Two days, single track  

The venue was a big industrial hall, the kind of place that fits a conference of 
this size while still feeling intimate enough to bump into the speakers between talks. Krakow itself was a great
backdrop: the old town, the main square, and the river walk were all within reach during after the conf (eg. I had 
the chance to visit Wavel Castel and Rynek Główny).

Now let's dive into the talks.

![The main stage at App.js Conf 2026, kickoff slide](/media/content/blog/post/2026/06/01/app-js-conf-2026/app-js-conf-2026-opening.jpeg "The main stage at App.js Conf 2026, kickoff slide")

## Day 1: Keynotes and incredible talks

The first day started, unsurprisingly, with two keynotes: one by Software Mansion and another one by Expo. 
After them a series of talks from "famous devs" from the react native scene took place. In particular I finally had a chance to see 
William Candillon live.

### Software Mansion Keynote

The software mansion keynote was held by [Krzysztof Magiera](https://x.com/kzzzf), director of engineering at Software Mansion.
In his keynote, he made a small introduction about how AI is changing the way we write software, speeding up 
coding writing and moving the focus to building, and testing. 
He stated that the work done by Software Mansion during the last years in the RN ecosystems put the company in the perfect spot 
to be the driver of this new way of working.

He started show all the contributions made by the company in the open source space.  He also mentioned they have a 
special partnership with Meta that allow them to be more proactive in their contributiuon to React Native itself. 
This led them to 283 commits in the last year on the React Native repository, with a strong contribution to the animation layer where they 
brought back into react native some idea originally developed in Reanimated, the famous library they maintain.
They also made strong contributions to Hermes V1.

They also made 752 commits on Expo repo. they contributed in particular to:

- expo browfield
- expo widget
- expo modules core

![Expo brownfield](/media/content/blog/post/2026/06/01/app-js-conf-2026/app-js-conf-2026-expo-brownfield.jpeg "Expo brownfield")

They are also actively continue to work on their own famous libraries:

- Reanimated, recently reached version 4.4, with support for css selector. They also extracted a few versions ago 
the worklets packages, so that it can be used in other context

![Reanimated update on stage at App.js Conf 2026](/media/content/blog/post/2026/06/01/app-js-conf-2026/app-js-conf-2026-reanimated.jpeg "Reanimated: CSS animations, Worklets, Core Animation, and CSS pseudoselectors")

- React Native screens, with support now for split views (iOS)
- Detour, a deep link platform (can it replace appsflyer)
- TypeGPU, presented in an ad-hoc talk that we will see later
- enriched, a set of libraries for rich text rendering like markdown (should we try it at lastminute.com?)
- RNRepo: set of prebuilds of famous libraries to speed up the build (should we try it at lastminute.com?)

He also present a new AI tool called [Argent](https://github.com/software-mansion/argent). 
It is a An agentic toolkit to control, debug, and profile iOS and Android apps. Using Argent, the llm can inspect the view hierarchy, access the debugger and profiler to debug bugs.

![Argent on stage at App.js Conf 2026](/media/content/blog/post/2026/06/01/app-js-conf-2026/app-js-conf-2026-argent.jpeg "Argent: an agentic toolkit to control, debug, and profile iOS and Android applications")

He also announced on stage version 3 of Gesture Handler. It is a new major version created to support the new architecture with a lot of improvement under the hood.

![Gesture Handler 3.0 on stage at App.js Conf 2026](/media/content/blog/post/2026/06/01/app-js-conf-2026/app-js-conf-2026-gesture-handler.jpeg "Gesture Handler 3.0: hook API, native touchables, and quality-of-life improvements")

Last but not least, they presented a new bottom sheet library. It has been presented as a bottom sheet "that just works" :laughing:.

![Software Mansion Bottom Sheet on stage at App.js Conf 2026](/media/content/blog/post/2026/06/01/app-js-conf-2026/app-js-conf-2026-bottom-sheet.jpeg "The new @swmansion/bottom-sheet, built on native primitives")

Next there was the Expo Keynote, held by [Charlie Cheever](https://en.wikipedia.org/wiki/Charlie_Cheever), CEO of Expo. 
He started by showing how react, react native and expo are blowing up in downloads charts. He also shown how RN apps are the one generating more revenues. 

"React native and Expo are winning the AI era", this was a statement he made. This is a consequence of the of the fact that React Native and Expo have a big community, with thousand of open source 
libraries and apps the LLMs has been trained on. Also the documentation expo has, combined with the support for all the major platforms (iOS, Android and Web).

He mentioned agan argent and agent device as the missing piece to close the feedback loop with AI. Expo should focus on web support, that is more important than ever. 
He then started shown how Expo is for real universal, starting to support different platforms like Expo TV, that adds the support for Apple TV and Android TV. 
Additionally he shown Expo UI, a library that contains a set of native components for iOS and android built with SwiftUI and Jetpack Compose, and expo-widget and expo-router

Then he shown some tools created by Expo like Observe, that we will see in detail in a talk I attended on day 2.
Most of the stuff presented above that is not a standalone product or library, is already available in Expo SDK 56.

### Pixelicious

Next, I finally had a chance to see William Candillon on stage for real. For me, he is like a legend. He is the developer I would like to be.

![William Candillon at App.js Conf 2026](/media/content/blog/post/2026/06/01/app-js-conf-2026/app-js-conf-2026-william-candillon-hello.jpeg "William Candillon at App.js Conf 2026")

He started by showing how the react native skia ecosystem is growing fast and how some of the most famous open source contributors and dev content creators 
are already experimnenting with react native webgpu, introduced last here at appjs conf.
He also shown Redraw, a new library for 2D rendering that he has recently presented.

He then started to show what he is working on for react native webgpu: aligning the api so that the code can be "copy-pasted" from the web to the native side. 
For example there was until now some difference in how you start the rendering loop in react native webgpu compared to what you had to do in the web, with an additional 
`context.present()` call to notify the native UI thread of the start of the loop. Now they are aligned.

![React native skia, webgp and redraw](/media/content/blog/post/2026/06/01/app-js-conf-2026/app-js-conf-2026-william-candillon-webgpu-skia-redraw.jpeg "React native skia, webgp and redraw")

He than show a new feature of react native worklets called [bundle mode](https://docs.swmansion.com/react-native-worklets/docs/bundleMode/) allowing devs to execute 
arbitrary code from your bundle on any thread. This unlock a lot of possibility, like having multiple render executing at the same time (2D and 3D). 

He also shown the work do with [Marc Rousavy](https://github.com/mrousavy), CEO of Margelo and author of [react-native-vision-camera](https://github.com/mrousavy/react-native-vision-camera), to have 
video and camera streaming plug and play into react native webgpu.

He then switched to react native skia, where he is developing a new backend called Skia Graphite, backed by Dawn Webgpu, the same implementation of webgpu that powers react native webgpu. 
This means that you can have view that draw using both react native webgpu and react native skia, because their runtime engine is the same. The plan for Skia Graphite is to have it 
used by everyone by Q4 of 2026.

He finally introduced redraw, a new library that can work with skia and webgpu, that uses typegpu under the hood to enhance the drawing capabilities by passing geometry and 
fragment shaders as function to the library components.

He closed the talk revealing that his two recent projects started from his tradition to create April's fool videos: 

- April's fool 2024 -> react native gpu
- April's fool 2025 -> redraw

He show this to leave with a quote that I quite like: "unproductive times never fail to end up being the most productive one". 

### How to Build the Fastest Apps: Break the Rules

Next in line was [Jay Meistrich](https://github.com/jmeistrich), with a presentation of his two most famous libraries: [Legend List](https://github.com/LegendApp/legend-list) 
and [Legend State](https://github.com/LegendApp/legend-state). This was quite an interesting talk, because the libraries presented could have bring a lot of benefits into 
a lot of apps.

LegendList is a list library, probably the fastest on the market. He announced version 3 release during the talk.
Then he started the real talk: making app really fast. So fast that they can beat native apps. He did a lot of consultancy work in the last here 
and he so the same problem in multiple code bases: unnecessary rendering.
The problem is that rendering is used also to coordinates effects and sharing information. React state is oversubscribed, and it is the orchestrator of everything. 
The solution that Jay propose is to rethink how we think about state, separating the ownership from the subscription to changes. 
He made some comparison with React, that call this concept signals, or Reanimated with shared value. 

This is why he created LegendState: the solution is to have observable, and the components can read the value of them when needed. This is the same approach used in 
other frameworks like Svelte and Preact.
He also made some considerations about legend state and how to convince yout team to use an alternative state library, showing how for most of the basic components 
of react native we are already using solution from the community and not the standard components.

![Why should I change my state library?](/media/content/blog/post/2026/06/01/app-js-conf-2026/app-js-conf-2026-jay-meistrich-legendstate-builtin-components.jpeg "Why should I change my state library?")

He then closed the talk reiterating how much important state architecture is, probably the most important architectural pattern for a react/react native app.

### Write TypeGPU, Use Anywhere: Composable WebGPU Shaders for React Native and Beyond

The afternoon started with another great talk by [Iwo Plaza](https://github.com/iwoplaza), software engineer at Software Mansion.
He presented a enw library they developed called [TypeGPU](https://docs.swmansion.com/TypeGPU/). 
He show how this library makes the developer life easy by writing shaders programs using TypeScript. He started from a demo of
react native webgpu to show how easy it is to yo write shader as callback using the directive `use gpu`. 

Why a new shader system? The standardize how to write shaders fro different engines, without sacrifing the type safety from TypeScript. 
By choosing TypeScript they were also able to use a unified set of building tools with respect to the ecosystem.

They also had to create [tsover](https://github.com/software-mansion/tsover): a "fork" of TypeScript that add operator overload to the language. 
This enhance the shading writing experience a lot (eg. simplify array operation a lot).

He also mentioned that TypeGPU is not only about computer graphics: it is also ready to do massive computation on the GPU.
Obviously, typeGPU has already its own official react integration, [@typeGPU/react](https://docs.swmansion.com/TypeGPU/ecosystem/typegpu-react/).

He concluded the talk with the slogan "Math is cool!" :heart::laughing:

## Day 2

On day 2 there was a mix of talks about different topics. I also had the chance to speak with a lot of people from different companies, to understand their ways of working.

### Expo on Desktop

The day started with a talk held by [Jamie Birch](https://github.com/shirakaba), a Software Developer based in Japan. 
He introduced a new open source project he developed: [Expo Desktop](https://github.com/shirakaba/expo-desktop).

He started with an overview of the current state of React Native on Desktop, where there are ad hoc frameworks for react native on different 
platform: react native macos and react native windows. Unfortunately, generally speaking, the ecosystem on these platform in terms of packages is smaller compared to 
react native on mobile. Additionally, the variants for macos and windows are usually behind in terms of updates related to react native, pinpointing some version behind the main one.
Anyway, especially for macOS, 1/3 of the packages available on iOS are already compatible. Also Expo UI it is already compatible without any change.

Expo Desktop is a toll to make Desktop apps first-class on Expo:

- by handling the setup for creating new app matching the user experience on mobile
- by implementing Expo Prebuild
- by filling missing parts of the ecosystem

He also stated that the framework is not vibe coded :laughing:. 
How does it work:

- he ported the config plugins, creating ad hoc variant per system
- he ported the cli tooling (`npx expo-desktop`)
 - the cli checks the current common version between react native macos and react native windows

He closed the talk by showing the official repo, and expressing the hope the project gets upstream in terms of usage and in terms of contributors.

### React Native without Cocoapods

Next in line there was [Riccardo Cipolleschi](https://github.com/cipolleschi) from Meta and [Christian Falch](https://github.com/chrfalch) from Expo showing how react native team with the help of expo 
is migrating away from Cocoapods to Swift Package Manager. 

First, Riccardo show the timeline until today of the work done to enable SPM in React Native. He also explained the reason why they are doping this: Cocoapods 
will be abandoned by the end of the year. They started to work on this in January 2025. The first step is related to the precompiled binaries for react native released some time ago. 
These are already built using SPM in the internal react native CI. 

Next a public RFC has been published by Christian Falch. This RFC explain the reason why they are doing this (reported above) and the plan on how to do it:

- ship SPM support behind a opt-in flow
- add CLI and template support so that apps can be created or migrated with official tooling
- keep support for both CocoaPods and SPM in parallel for some releases
- define a clear contract for third party libraries and encourage them to ship dual support
- move SPM to the default when ecosystem is ready

Then Christian shown a live demo of how it will work the integration. The main command will be `npx react-native spm`. 
The command will add the Package.swift for the deps that still don't have it and will try to rename the old xcodeproj by adding it a `.legacy` extension, so that our app will be loaded using
 the SPM app project. Opening a project after running this command will show 4 packages:

 - Autolinked, for native packages automatically discovered and linked 
 - React-GeneratedCode, this is a codegne library
 - ReactNative, this contains the precompiled binary mentioned at the beginning
 - Plus, an entry for each library automatically scaffolded by the command above

As soon as the setup is done, adding new libraries is easier than before: we just install a library through npm, and if it has a `Package.swift` 
it will be added automatically as a dep on the native side. 

For libraries author, the process is quite simple too. There is the `Package.swift` to add (can be scaffolded with the command above, but christian didn't share any details), 
and then add a declaration to the react-native.config.js file (in a new `spm` section). At the beginning libraries can skip the step to provide a precompiled binary.

### One Codebase, Three Platforms: Making Universal Expo Work in Production

The next talk was from [Edwin Vargas](https://www.linkedin.com/in/edwinvrgs/) from [handoff.ai](https://handoff.ai), showing how his team his trying to make e universal app 
with Expo.

He started with an introduction of how he discovered Universal Expo. They started the company in 2023, and they were trying to scale quickly. This is a consequence of 
the fact that they needed speed to find their market fit. They started from the *mobile app*. Then they deployed to the web. This didn't come without challenges.

First of all, they were leveraging a lot Facebook/Instagram Ads. This means their web application was goingh to be shown in a non standard browser, especially on android. 
This is when they discovered an hard true: mobile web is the 4th platform (after web desktop, iOS app and android app). To overcome some of the problem, they tried 
to push out from the Meta in app Browser top their app the users when they were signing up. they discovered this was bad for conversion. 

They leveraged a lot OTA updates on mobile (EAS) to keep the mobile apps aligned with the web experience. 

One thing that is really interesting, is that even if the codebase is one for all the platforms, they pushed a lot to have platform differences supported in the right way 
at the right time. For examples, they used libraries for mobile when needed and for web in the other cases. They also pushed a lot to create the perfect design system for their needs 
to support each platform in the right way.

He condluded the talk with a question "Did the universal expo bet pay off?". The answer was yes because:

- they built complex screen sharing lot of the code 
- they speed up the development a lot 
- they were also able to scale the team, from start up to scale up size
- they were able to learn from customer faster than ever given the fast release track on all platforms

### Accessibility in Motion

Then it was the time for [Britta Evans-Fenton](https://x.com/13ritta), senior developer at from Shopify. 
She made a talk about accessibility in apps.

She started asking to the audience how many accessibility features used in the last week? Some of them seems granted/not accessibility related, but they 
have been created for this purpose. She then introduced the concept of [curb cut](https://en.wikipedia.org/wiki/Curb_cut_effect) effect, a the phenomenon of 
disability-friendly features being used and appreciated by a larger group than the people they were designed for. She also introduced the concept of [universal design](https://en.wikipedia.org/wiki/Universal_design), 
that is the design of buildings, products, or environments to make them accessible to all people, regardless of age, disability, or other factors. The general point 
that connects these concepts is: if you design for the edge, you are building for everyone.

She then started by saying she didn't want to cover the basic API contained in React Native (accessibility props on components). Also, given AI this day,
for react native not everything is covered like web, because apps tend to be less accessible with respect to the web. 

She then explained why she is so sensitive about accessibility: when she started to work on react native she discovered that she has the same gene of her grand mother that lends to 
macular degenration, a disease that affect the central vision making you partially blind. This is why she started to be a promoter to make apps more accessible.

She then started with an overview of how we can make apps more accessible. The first wa reduce motion, something that could help people with ADHD, vestibular disorders, 
photosensitive episelpsy. She then shown the api we can use to understand if this accessibility feature is on, and tweak the animation based on its values. 
Nice thing, this is an API where I made some small contribution on the React Native project.

![React native accessibility reduce motion API](/media/content/blog/post/2026/06/01/app-js-conf-2026/app-js-conf-2026-accessibility-reduce-motion.jpeg)


She then switched to color accessibility, where she explained how important is to have a clear difference between colors that represents different states. 
She then talk about screen readers. She explained how to tackle cases where pieces of UI appear later on. this should be managed with ad hoc api to set the focus on the right element 
after and interaction.
She finally closed the talk with some consideration and possible action to manager differen UI components (Toast, Gesture Swipe etc.).
