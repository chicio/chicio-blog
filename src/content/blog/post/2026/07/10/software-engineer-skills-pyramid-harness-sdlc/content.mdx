In the introduction we speak about the fact that lastminute chose Anthropic and the claude models as our standard AI. 
I want to speak about why we are creating a claude code plugin:
- Experiment with AI, the new technology that is disrupting our field
- we want to automatize and encode in claude code skills and agent our "software factory": how we are used to develop software encoded in skills and agents
-- we developed our harness as a plugin for claude code
This project is an experiment. We are trying to understand how much we can push in automatize our workflows: the tradeoff between automatisation and benefit in doing it. This is because not everything is useful to be automatize: this is where real engineering and value for the company shows up.
Also in terms of how to write good skill, it has been (and still it is) a try and learn journey: we learn from our mistakes and attempts.

## Skills vs Agents: a quick comparison

A brief comparison between what is an agent and what is an agent.

- Agent: WHO does the work: an isolated worker with its own context window, own tool whitelist, own model, own persona. It decides how to reach the goal (it's not a fixed script). Caller only sees its final message.
- Skill: HOW to do a task: on-demand procedural knowledge (a SKILL.md + optional scripts) that loads into whoever invokes it — same context, not a new one. No own model/tools.

## Our workflow

Before jumping in the plugin description, I woould like to talk about the state of the art of our ways of working. I will speak about how we leverage SCRUM as a workflow for our software development lifecycle:
- request from stakeholders comes in
- the product manager brings to the refinement the request, with or without the user story
- the team with the product manager, analise during refinement the request and produce a user story
- after the refinement, the team takes the story and develop it
- the devepment produces a pull request to be shared with other developers for code review (and code validation)
- after approval, the new feature gets deployed in QA for product validation, to e2e test that everything works correctly
- after e2e approval, the feature goes to production

## Our claude code plugin

We tried to encode in skills and agents the ways of working described above.
We encoded the full process.

The first step is thw write of the jira story, in a skill called `write-jira-user-story`. We created a skill ad hoc to do this. It brings a specific layout to write them made as follow:

- title
- acceptance criteria
- tech todo
- validation

Among this, propably the most relevant part is the tech todo section. here we try to be as precise as possible with respect to the actual code structure: in this way we try to guide the next phase to point the LLM already to the right direction in terms of design and implementation of the feature.
The ticket wrote in this way is important not only for the our workflow integration but also to act as documentation of the feature (so we can consult it if we need to understand the reason why code contains some logics)
The skill uses the [atlassian plugin](XXX link to plugin) to interact with our boards and create the story already in the backlog.

The second step is the implementation of the story. For this we have 3 specific skills, one for each tech stack in our company Backend, frontend web and mobile, that uses specific agents:

- `sdlc-mobile`
- `sdlc-backend`
- `sdlc-frontend`

The skills have each one their own peculiarity in terms of tech stack. For example, the mobile one is focused on react native and related technology. The backend one is focused on spring boot + java/kotlin + our own internal abstraction over spring boot, [appfw](XXX link to the article about BE and Frontend technology).  
Even if with differences related to the tech stack, the share a common workflow. The skill act as an orchestrator of the SDLC, software development lifecycle, delegating to subagents specific operations based on the phase where we are.

All these skills accepts a set of shared parameters, each one triggering an ad hoc function:

- `<jira ticket>`, the user story to develop
- `<figma url>`, the url of the mockup. This is a parameter accepted only by the FE skills (`sdlc-frontend` and `sdlc-mobile`)
- `--isolated`, to start a workflow in a `git worktree`, isolated from the main one. This let us trigger multiple workflow on the same codebase.

Let's see the common workflow the skills share between each other.

### sdlc-e2e

All these skills above are invocable manually on a specific project. Anyway we tried something more ambitious: we create a specific skill to orchestrate all of them, capable of accepting a generic user command like `write jira with this content <description of the task>`, or `implement this jira ticket: <ticket id>`. This skill will select, in case of implementation, the right skill for the tech stack of the project you are into. 
It will also be able, for the backend tech stack, to deploy the result of the code to our microservice leveraging another skill called `be-deploy`, that uses under the hood the [argo mcp](XXX Link) (falling back to [`argocli`](XXX LINK) when not available) to monitor the progress of the deploy.

## Plugin Architecture overview

Below you can find a full overview of the architecture of the plugin, in terms of interaction between skills and agents. This is a recap of the full plugin architecture.
As you can see, we came up with a solution that includes different layers of abstractions: at the upper levels there are skills that interact directly with a human input, and the only purpose is to translate what they receive in a more technical language (see the e2e skill). At the middle levels, skills are more specific and has a more techincal knowledge (see the sdlc-mobile, be and fe). At the lowest levels, there are agents that are the operative layer that interact with the specific technology (see the implementer that, guided by the plan done before by the explorer, writes the actual Kotlin/Typescript code, or the dev-ops that checks Gitlab CI status using Gitlab APIs). This is a design approach that mymics other engineering structures we find in software engineering (think about the test pyramid, or the divide et impera approach of the procedural programming and what we are used to see in the Object Oriented Programming).

## Improvements

The plugin is far from perfect. We already noted some possible improvements:

- speed. The plugin takes more time in terms of execution, especially in the implementer vs code reviewer phase. We are thinking about possible solution like trying to parallelize tasks implementation.
- micro commit approach. Sometimes the implementer agent tends to push to much the micro commit approach. We are still thinking about possible solutions to improve this.
- validation. We would like to push more the validation phase, to generate also video of the QA phase that could be reviewed by us.

## Conclusion and next step

The plugin is still in an experimental phase. We are testing it in our daily job, tweaking it as soon as we find out problems and bugs.
In our opinion, up to now the results were pretty good and quite impressive. We were able, giving a request from our product manager, to completely automatize the workflow for specific tasks while we were focusing on much more harder and "human related" work: system design, product discovery, code review of other features, manual deep validation of the product.  
One think we would like to say, is that for sure, a tool like this, reduced our time to market for releasing new features, and we noticed it in our day by, but unfortunately, at the moment we don't have a specific metric to demonstrated the effectiveness of this plugin.  
Also because, related for example to the metrics recently shared by Spotify, we don't belive that measuring the number of deploy if a relevant metric for assessing the goodness of the AI tools.  
We will try to understand in a next phase how to understand the real impact.
